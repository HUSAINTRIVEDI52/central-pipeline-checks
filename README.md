# 🔐 Central Pipeline Checks


## What This Does

Every time code is pushed to this repository, the pipeline automatically:

1. **Scans source code** with SonarQube (SAST) — finds insecure coding patterns, hardcoded secrets, and code quality issues
2. **Scans dependencies** with OWASP Dependency-Check (SCA) — finds npm packages with known CVEs
3. **Imports findings** into DefectDojo — a central dashboard to track, assign, and resolve vulnerabilities
4. **Uploads a report** as a GitHub Actions artifact — downloadable for 30 days

| Tool | Type | What it finds |
|------|------|---------------|
| SonarQube | SAST (Static) | Insecure code patterns, hardcoded credentials, XSS risks |
| OWASP Dependency-Check | SCA (Dependencies) | npm packages with known CVEs from the NVD database |
| DefectDojo | Aggregation | Central dashboard to manage all findings across scans |

---

## Pipeline Overview

The pipeline is defined in `.github/workflows/security-pipeline.yml` and runs on a **self-hosted GCP VM**.

```
Trigger (push / PR / cron / manual)
         │
         ▼
  ┌─────────────┐
  │  Job 1      │  Set commit SHA, date, branch metadata
  │  Prepare    │  Auto-install Docker + Node.js if missing
  └──────┬──────┘
         │
         ▼
  ┌─────────────────────────────┐
  │  Job 2 — Run Scans          │
  │  ┌──────────────────────┐   │
  │  │  Step 1: SonarQube   │   │  Static code analysis
  │  │  SAST                │   │
  │  └──────────────────────┘   │
  │  ┌──────────────────────┐   │
  │  │  Step 2: OWASP       │   │  Dependency CVE scan
  │  │  Dependency-Check    │   │
  │  └──────────────────────┘   │
  └──────┬──────────────────────┘
         │
         ▼
  ┌─────────────────────┐
  │  Job 3              │  Import to DefectDojo
  │  Collect Report     │  Upload artifact to GitHub
  └──────┬──────────────┘
         │
         ▼
  ┌─────────────┐
  │  Job 4      │  Remove containers + temp files
  │  Cleanup    │
  └─────────────┘
```

## Prerequisites

> ⚠️ **This pipeline requires a self-hosted runner.** GitHub's default hosted runners do not work here — the tools are too heavy and need persistent storage for the NVD cache.

You need the following on your GCP VM before setting up the pipeline:

- [ ] A GCP VM (see specs below)
- [ ] Docker installed
- [ ] Node.js v16+ installed
- [ ] GitHub Actions self-hosted runner configured

---

## Tool Setup

### 1. GCP VM (Self-Hosted Runner)

**Recommended specs:**

| Setting | Value |
|---------|-------|
| Machine type | `e2-standard-4` (4 vCPU, 16 GB RAM) or larger |
| Disk | 50 GB SSD (NVD cache alone is ~3–5 GB) |
| OS | Ubuntu 22.04 LTS |
| Firewall | Allow inbound TCP on port `8080` (DefectDojo) and `9000` (SonarQube) |

**Install Docker:**

```bash
sudo apt-get update
sudo apt-get install -y docker.io
sudo usermod -aG docker $USER
sudo systemctl start docker && sudo systemctl enable docker
```

**Install Node.js 18:**

```bash
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt-get install -y nodejs
```

> The pipeline will auto-install both if missing, but doing it manually first avoids first-run delays.

---

### 2. GitHub Actions Runner

The runner is the agent on your VM that listens for jobs from GitHub.

1. Go to your repo → **Settings** → **Actions** → **Runners** → **New self-hosted runner**
2. Select **Linux** → **x64**
3. Run the commands GitHub provides on your VM:

```bash
mkdir actions-runner && cd actions-runner

# Download the runner (GitHub will give you the exact URL and token)
curl -o actions-runner-linux-x64-2.332.0.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.332.0/actions-runner-linux-x64-2.332.0.tar.gz
tar xzf ./actions-runner-linux-x64-2.332.0.tar.gz

# Configure with your repo URL and token from GitHub
./config.sh --url https://github.com/YOUR_ORG/YOUR_REPO --token YOUR_TOKEN

# Install and start as a system service
sudo ./svc.sh install
sudo ./svc.sh start
```

> A pre-downloaded runner tarball is available in `actions-runner/` for convenience, but always use the token and URL generated by GitHub for the `config.sh` step.

---

### 3. SonarQube

SonarQube performs static code analysis (SAST). Run it as a Docker container on the same VM.

**Start SonarQube:**

```bash
docker run -d --name sonarqube \
  --restart unless-stopped \
  -p 9000:9000 \
  -v sonarqube_data:/opt/sonarqube/data \
  -v sonarqube_logs:/opt/sonarqube/logs \
  sonarqube:community
```

Wait ~2 minutes, then open `http://YOUR_VM_IP:9000`. Default login: `admin` / `admin` (you'll be prompted to change it).

**Create a project and get a token:**

1. Click **Create Project** → **Manually**
2. Enter a **Project Key** (e.g. `localit-backend`) — note this for `SONAR_PROJECT_KEY`
3. Go to **My Account** (top-right avatar) → **Security** → **Generate Token**
4. Name it `pipeline-token`, select **Project Analysis Token**, choose your project
5. Click **Generate** and copy the token immediately — this is `SONAR_TOKEN`

> Use `http://localhost:9000` as `SONAR_HOST_URL` if SonarQube runs on the same VM as the runner. Do not use the external IP — keep traffic internal.

---

### 4. NVD Database

The pipeline uses a pre-existing NVD database on the GCP instance instead of downloading it.

**Setup the NVD database:**

```bash
# Create the NVD database directory
sudo mkdir -p /home/runner/setup-pipeline

# Download and extract the latest NVD database (one-time setup)
wget https://github.com/jeremylong/DependencyCheck/releases/download/v8.4.0/nvd-data.zip
sudo unzip nvd-data.zip -d /home/runner/setup-pipeline/
sudo mv /home/runner/setup-pipeline/nvd-data/* /home/runner/setup-pipeline/
sudo chown -R $(whoami):$(whoami) /home/runner/setup-pipeline/

# The database file should be at: /home/runner/setup-pipeline/nvd_database.json
```

> The pipeline will fail if `/home/runner/setup-pipeline/nvd_database.json` does not exist.

---

### 5. DefectDojo

DefectDojo is the vulnerability management dashboard. All scan findings are imported here.

**Install with Docker Compose:**

```bash
git clone https://github.com/DefectDojo/django-DefectDojo.git
cd django-DefectDojo
docker compose up -d
```

Wait ~5 minutes, then open `http://YOUR_VM_IP:8080`.

**Get the admin password:**

```bash
docker compose logs initializer | grep 'Admin password:'
```

**Create a Product:**

1. Go to **Products** → **Add Product** → fill in a name → **Submit**
2. Note the numeric ID in the URL: `/product/3/` → ID is `3` → this is `DEFECTDOJO_PRODUCT_ID`

**Create an Engagement:**

1. Inside your product → **Engagements** → **Add Engagement**
2. Name it (e.g. `CI Pipeline`), set Status to **In Progress** → **Submit**
3. Note the numeric ID in the URL: `/engagement/7/` → ID is `7` → this is `DEFECTDOJO_ENGAGEMENT_ID`

**Get your API key:**

1. Click your username (top-right) → **API v2 Key**
2. Copy the key — this is `DEFECTDOJO_API_KEY`

> Use `http://localhost:8080` as `DEFECTDOJO_URL` if DefectDojo is on the same VM as the runner.

## GitHub Secrets

All sensitive values are stored as encrypted GitHub Secrets — **never committed to the repository**.

**To add secrets:** Repo → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**

| Secret | Example Value | Where to get it |
|--------|---------------|-----------------|
| `SONAR_HOST_URL` | `http://localhost:9000` | Your SonarQube URL (use `localhost`) |
| `SONAR_TOKEN` | `sqp_abc123...` | SonarQube → My Account → Security → Token |
| `SONAR_PROJECT_KEY` | `localit-backend` | The key you set when creating the SonarQube project |
| `DEFECTDOJO_URL` | `http://localhost:8080` | Your DefectDojo URL (use `localhost`) |
| `DEFECTDOJO_API_KEY` | `1a2b3c4d5e...` | DefectDojo → Username → API v2 Key |
| `DEFECTDOJO_ENGAGEMENT_ID` | `7` | Numeric ID from the engagement URL |
| `DEFECTDOJO_PRODUCT_ID` | `3` | Numeric ID from the product URL |

> ⚠️ Never put these values in `.env`, YAML files, or anywhere in the repository.

## Running the Pipeline

### Automatic

Push any code change to `main`, `develop`, or a `feature/**` branch and the pipeline starts automatically.

### Manual

1. Go to your repo → **Actions** tab
2. Select **Security Scanning Pipeline** on the left sidebar
3. Click **Run workflow** → **Run workflow**

### Downloading the Report

1. Open any completed workflow run in the **Actions** tab
2. Scroll to the bottom → **Artifacts**
3. Download the ZIP named `security-report-BRANCH-SHA-DATE`
